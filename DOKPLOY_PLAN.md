# Ads MCP — Dokploy Deployment & Repo Comparison Plan

Scope: decide which MCP server to run on Dokploy for AI‑driven Google Ads (and later Meta/GA4) campaign management, and get it running behind `https://ads-mcp.rahulverma.cc`.

---

## 1. Repo: `googleads/google-ads-mcp` (official) — forked at `RahulVerma989/google-ads-mcp`

### What it is
- Apache‑2.0, maintained by Google, Python 3.10+, built on FastMCP 3.2+ and `google-ads` (Ads API v23).
- Package name `google-ads-mcp`; entry point `ads_mcp.server:run_server` in `pyproject.toml:32`.

### Tools exposed (the entire surface)
- `search` — `ads_mcp/tools/search.py` — runs GAQL via `GoogleAdsService.search_stream`. Read‑only.
- `get_resource_metadata` — `ads_mcp/tools/get_resource_metadata.py` — resource field discovery. `readOnlyHint=True`.
- `list_accessible_customers` — `ads_mcp/tools/core.py` — lists customer IDs reachable by the auth principal.

### Resources exposed
- `discovery-document`, `metrics`, `segments`, `release-notes` — all static/discovery helpers.

### Auth
- `ads_mcp/coordinator.py` toggles transport automatically:
  - If `GOOGLE_ADS_MCP_OAUTH_CLIENT_ID` + `_SECRET` are set → `FastMCP(..., auth=GoogleProvider(...))` and server starts with `transport="streamable-http"`.
  - Else → stdio.
- OAuth proxy scopes include `https://www.googleapis.com/auth/adwords` (which is actually a read‑write scope — the server just never uses mutations).
- `utils._create_credentials()` prefers the FastMCP access token when present, falls back to ADC.

### Deployment
- Ships as a pip package. Your current Dokploy compose runs `pip install git+...` and then `google-ads-mcp` with `FASTMCP_HOST/PORT=0.0.0.0:8000` → fine. The MCP endpoint is at `/mcp`.
- The OAuth redirect lives at `{GOOGLE_ADS_MCP_BASE_URL}/auth/callback` (FastMCP GoogleProvider default) — must be added to the OAuth client's authorized redirect URIs in GCP.

### Strengths
- Tiny, auditable, official, well‑aligned with MCP spec.
- FastMCP OAuth proxy means each Claude/ChatGPT user logs in as themselves — your server doesn't hold their tokens.
- Descriptions are GAQL‑aware (`search.py:_search_tool_description`) which makes LLMs quite good at building queries.

### Weaknesses / bugs
- **Read‑only. No mutate tools at all.** You explicitly can't: create campaigns, pause/resume, edit budgets, add keywords, upload conversions, attach extensions.
- Hardcoded to Google Ads API v23 types (`ads_mcp/utils.py:22`, `search.py` hints); bumping API versions is a manual edit.
- `get_resource_metadata` has fallback branches that silently swallow errors (`utils.logger.warning`) — an LLM can end up with an empty selectable list and build invalid queries.
- `gaql_resources.txt` is generated by `update_references.py`. Stale checkouts ship stale grammar; the `_search_tool_description` only warns "Tool may not function correctly."
- No tests for the OAuth HTTP path; 0 open issues / 0 PRs on your fork (I checked), but the issue tracker is on the upstream `googleads/google-ads-mcp`.
- Implicit assumption: login‑customer‑id applies to all calls; no per‑tool override.

### Improvements I'd make if we adopt this
1. Add a `mutate` family: `update_campaign_status`, `update_campaign_budget`, `add_negative_keywords`, `pause_ad_group`, `create_search_campaign`, `upload_offline_conversions`. Pattern mirrors `search.py` — get the service, build operations, wrap `GoogleAdsException` in `ToolError`.
2. Surface a `customer_id` argument on every tool so one MCP session can hop between MCC children without restarting.
3. Add confirmation/guardrail: `mcp.tool(annotations=ToolAnnotations(destructiveHint=True))` on every mutate, plus a `dry_run` arg that calls `validate_only=True` first.
4. Fix the silent‑fallback swallow in `get_resource_metadata.py` — at minimum log at ERROR and return a "metadata incomplete" flag.
5. Cache `gaql_resources.txt` at startup and re‑generate on version bump (currently a separate CLI).
6. Dockerfile (the repo has none) + a `Dockerfile.dokploy` that avoids the `apt-get install git` inside `python:3.12-slim` on every container start.
7. Pin `google-ads`, `fastmcp`, `mcp` minor versions — currently unbounded upper bounds.

## 2. Repo: `RahulVerma989/google-meta-ads-ga4-mcp`

### What it is
- MIT‑licensed, but — **there is no source code in the repo**. Tree is `.github/`, `configs/`, `docs/`, `CHANGELOG.md`, `LICENSE`, `README.md`. Nothing else.
- README advertises "250+ tools" (Google Ads 150+, Meta 80+, GA4 20+) and a hosted MCP endpoint. Every setup example is literally `"url": "YOUR_MCP_ENDPOINT_URL"` — the real endpoint is the commercial service at https://www.get-ryze.ai/ (Ryze AI). The repo is a marketing/docs front‑end, not a deployable server.

### Conclusion on self‑hosting
- **You cannot run this on Dokploy.** There's no Dockerfile, no pyproject, no package, no server entry point to install — only JSON snippets that point a client at a third‑party URL you don't control.
- Adopting it means paying Ryze AI and routing all your Google/Meta/GA4 OAuth tokens through their servers. That's a real trust decision, not a self‑hosting decision.

### Strengths (of the hosted product it advertises)
- Big tool surface across three platforms, cross‑platform prompts ("compare Google vs Meta ROAS").
- OAuth flow handled for you; setup is a single URL in the client config.
- Good docs: `docs/SETUP_CLAUDE.md`, `docs/SETUP_N8N.md`, `docs/EXAMPLES.md` are solid references for prompts you'd want to run.

### Weaknesses / risks
- Closed source — you can't audit what's sent with a mutate call, can't pin versions, can't patch bugs.
- Your Google Ads + Meta + GA4 tokens end up on Ryze's servers.
- README claims `250+ tools` with full write access but 0 code is reviewable. Claims are unverifiable from the repo alone.
- `CHANGELOG.md` shows one "1.0.0" entry and `list_branches` shows only `main` with 1 commit — it's a brand‑new marketing page.
- README lists `create_search_campaign`, `create_pmax_campaign`, etc., but because the spec only lives in docs you don't know their arg shapes, validation behavior, or rate‑limit handling.

### Improvements
- If we went down this path, the only real move is to vendor their hosted URL behind our own bearer‑authed proxy so at least we control network egress and can swap providers. That's a lot of plumbing for no code gain.

## 3. Side‑by‑side

| Criterion | `google-ads-mcp` (official fork) | `google-meta-ads-ga4-mcp` |
|---|---|---|
| Source available | Yes | **No source, docs only** |
| Self‑hostable | Yes (already works on your Dokploy) | No (hosted SaaS) |
| Platforms | Google Ads only | Google Ads + Meta + GA4 (hosted) |
| Write/mutate tools | **None** | Claimed yes, unverifiable |
| Auth | FastMCP OAuth proxy (per‑user) or ADC | OAuth via Ryze servers |
| License | Apache‑2.0 | MIT (on docs); hosted service ToS separate |
| Maintainer | Google | Ryze AI (commercial) |
| Fit for your goal ("AI creates/manages campaigns") | Needs write tools added | Fits out of the box, but on someone else's server |

## 4. Recommendation

**Ship the official `google-ads-mcp` on Dokploy now** (read path) and add the mutate tools we actually need as a second step. Concretely:

### Phase A — today, zero code change
1. Keep your current Dokploy compose for `google-ads-mcp`, with two hardening fixes:
   - Replace the `pip install git+...` on container start with a pinned image built once. Saves ~60s boot and removes `apt-get install git` at runtime.
   - Make sure `GOOGLE_ADS_MCP_BASE_URL=https://ads-mcp.rahulverma.cc` matches the Traefik Host rule exactly, and that `https://ads-mcp.rahulverma.cc/auth/callback` is in the GCP OAuth client's authorized redirect URIs.
2. Keep using it read‑only. Connect Claude via `httpUrl: https://ads-mcp.rahulverma.cc/mcp`.

### Phase B — fork and add writes
3. On `RahulVerma989/google-ads-mcp` (branch `claude/mcp-dokploy-deployment-5mLr9`), add a `ads_mcp/tools/mutate.py` with the minimum set to run paid acquisition:
   - `update_campaign_status(customer_id, campaign_id, status)` — pause/enable.
   - `update_campaign_budget(customer_id, campaign_id, amount_micros)`.
   - `add_negative_keywords(customer_id, campaign_id|ad_group_id, keywords[])`.
   - `create_search_campaign(...)` with `validate_only`/`dry_run` toggle.
   - `upload_click_conversions(...)` for the Routines that close the loop on ROAS.
   - Each mutate tool uses `ToolAnnotations(destructiveHint=True)` and wraps errors via `ToolError` like `search.py` already does.
4. Ship our own image: `ghcr.io/rahulverma989/google-ads-mcp:<sha>` built from a Dockerfile in the repo, referenced from Dokploy compose (no more git+pip on boot).

### Phase C — Meta + GA4
5. For Meta, run a **second** container on Dokploy using `pipeboard-co/meta-ads-mcp` (Python, FastMCP, supports write ops; has a Docker image). Wire it under `meta-mcp.rahulverma.cc`.
6. For GA4, run the community `surendranb/google-analytics-mcp` or equivalent under `ga4-mcp.rahulverma.cc`. (We can pick once Phase A/B is stable.)
7. Connect all three to Claude/ChatGPT; the LLM can now do the cross‑platform prompts that Ryze's marketing page advertises — on your infra.

Why this beats going with the Ryze repo:
- You keep ownership of tokens and mutation paths.
- You can audit/approve every destructive tool call in a PR before it ships.
- You're not locked into a SaaS bill scaling with your ad spend.

The only thing Ryze gets you that this doesn't is a unified endpoint day‑one. That's a few days of plumbing, not a product differentiator.

## 5. Dokploy compose (Phase A, drop‑in)

Minor improvements over your current compose: healthcheck, pinned Python, no more runtime `apt-get install git` by using `pipx` from a slim image that already has git baked in via a tiny install layer. If you want to stay with the exact same mechanic you have now, only the healthcheck block and `restart: unless-stopped` ordering matter.

```yaml
version: "3.9"
services:
  google-ads-mcp:
    image: python:3.12-slim
    restart: unless-stopped
    command: >
      sh -c "apt-get update &&
             apt-get install -y --no-install-recommends git &&
             rm -rf /var/lib/apt/lists/* &&
             pip install --no-cache-dir 'git+https://github.com/RahulVerma989/google-ads-mcp.git@claude/mcp-dokploy-deployment-5mLr9' &&
             google-ads-mcp"
    environment:
      GOOGLE_ADS_MCP_OAUTH_CLIENT_ID: ${GOOGLE_ADS_MCP_OAUTH_CLIENT_ID}
      GOOGLE_ADS_MCP_OAUTH_CLIENT_SECRET: ${GOOGLE_ADS_MCP_OAUTH_CLIENT_SECRET}
      GOOGLE_ADS_MCP_BASE_URL: ${GOOGLE_ADS_MCP_BASE_URL}   # https://ads-mcp.rahulverma.cc
      GOOGLE_PROJECT_ID: ${GOOGLE_PROJECT_ID}
      GOOGLE_ADS_DEVELOPER_TOKEN: ${GOOGLE_ADS_DEVELOPER_TOKEN}
      GOOGLE_ADS_LOGIN_CUSTOMER_ID: ${GOOGLE_ADS_LOGIN_CUSTOMER_ID}
      FASTMCP_HOST: "0.0.0.0"
      FASTMCP_PORT: "8000"
    expose:
      - "8000"
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.google-ads-mcp.rule=Host(`ads-mcp.rahulverma.cc`)"
      - "traefik.http.routers.google-ads-mcp.entrypoints=websecure"
      - "traefik.http.routers.google-ads-mcp.tls.certresolver=letsencrypt"
      - "traefik.http.services.google-ads-mcp.loadbalancer.server.port=8000"
```

Checklist before first boot:
- [ ] GCP OAuth client: add `https://ads-mcp.rahulverma.cc/auth/callback` as an Authorized redirect URI.
- [ ] Developer token has at least Basic access for the test account, Standard for prod.
- [ ] `GOOGLE_ADS_LOGIN_CUSTOMER_ID` is the MCC ID (digits only, no hyphens).
- [ ] Claude config: `{"mcpServers":{"google-ads-mcp":{"httpUrl":"https://ads-mcp.rahulverma.cc/mcp"}}}`.

Phase B adds a `Dockerfile` in this repo and swaps the `image:` line for `ghcr.io/rahulverma989/google-ads-mcp:<tag>`.

## 6. Open questions to resolve before Phase B

- Do you want a single MCC‑wide server, or a per‑client server (one container per customer)? Affects whether `customer_id` should be a required per‑tool arg.
- Are the Routines LLM‑driven (Claude deciding to pause a campaign) or deterministic (cron → GAQL → threshold)? Only the first needs mutate tools; the second can live outside MCP.
- For offline conversions: where's the source of truth for "paid user" — your DB? Stripe? That determines which `upload_*_conversions` variant we wire up.
